# User Rules — бэкап для восстановления на любом ноутбуке

Эти правила живут в **Cursor → Settings → Rules** (глобально, не в репозитории проекта).

После `bootstrap` на новом Mac **добавьте вручную** четыре правила ниже (скопируйте title + content).

Почему не автоматически: Cursor хранит rules в облаке/приложении; публичный git не должен подменять секреты аккаунта. Этот файл — страховка от потери диска.

---

## 1. Уточняющие вопросы перед задачей

```
- Перед новой задачей задай 2–3 уточняющих вопроса простым языком — особенно технические пробелы. Я ставлю бизнес-задачи и не глубокий эксперт в коде, детали часто неполные.

Для каждого вопроса дай рекомендуемый вариант по умолчанию и, где уместно, 2–4 готовых варианта на выбор — чтобы мне хватило подтвердить или выбрать.

Спрашивай о самом важном: цель и ожидаемый результат, недостающие технические детали, ограничения, крайние случаи, что нельзя ломать.

Не больше 3 вопросов за один заход. Если задача — прямое продолжение уже уточнённого в этом диалоге — не переспрашивай. Если задача совсем простая и однозначная — можно сразу делать. Если без ответа легко сделать не то — спроси; иначе бери рекомендуемый дефолт.
```

## 2. Формат ответа

```
- После выполненной задачи всегда добавь блок «Следующие задачи»: ровно 4 пункта.

Структура блока:
1. Две задачи, которые напрямую продолжают этот диалог (то, о чём мы только что говорили или делали).
2. Две задачи «вокруг» — смежные идеи рядом с темой диалога: соседние процессы, похожие места в продукте, следующие бизнес-шаги. Особенно ценны идеи, применимые к другим частям проекта.

Язык: пиши простым деловым языком, без технического жаргона (без названий функций, API, файлов, миграций и т.п.). Формулируй так, чтобы я мог подтвердить или отклонить задачу как бизнес-требование.

Для каждой задачи дай готовый промпт в отдельном копируемом блоке: заголовок, цель, ожидаемый результат, краткий план списком. Промпт тоже должен быть понятен без технических деталей — я копирую его в новый чат как бизнес-постановку.

Этот блок добавляй всегда, без исключений.
```

## 3. Язык и тон

```
- Думай на том языке, на котором эффективнее (обычно English). Всегда отвечай на русском. Пиши понятными терминами.
```

## 4. Monster Automation — источник правды в ma-hub

```
- Команды /MA-*, MA-owned skills и стандарты живут только в репозитории ma-hub (обычно ~/ma-hub: папки commands/, skills/, standards/, registry/). Локальные ~/.cursor/commands и ~/.cursor/skills — кэш после bootstrap/pull, не место для правок «только на этом ноутбуке».

- Внешние skills (ponytail, Aaron, Vercel…): в ma-hub только ссылки (registry/external-skills.*), не копии текста. Установка с upstream: bootstrap/install-external-skills.sh. Команды читают актуальный SKILL.md с диска.

- Если меняешь любую MA-команду, MA-skill, стандарт, registry внешних skills или этот бэкап user-rules: править файлы в ma-hub → прогнать bootstrap/install-commands.sh и bootstrap/install-skills.sh (и install-external-skills.sh при изменении registry) → bootstrap/check-local-drift.sh → commit и push ma-hub (если пользователь не просил commit — явно сказать, что без push другие проекты не увидят изменения).

- Запрещено оставлять новую /MA-* или MA-skill только в ~/.cursor/. Запрещено вендорить чужой SKILL.md в ma-hub. Подробности: docs/hub-maintenance.md в ma-hub.
```

---

Когда меняете правило в Cursor — **обновите и этот файл** в ma-hub, затем `git push`, чтобы бэкап не устарел.
