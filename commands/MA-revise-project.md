# /MA-revise-project

Подтянуть **текущий проект** к актуальным стандартам из публичного **ma-hub**.

Вы — консультант/БА. Ассистент — senior, который аудирует и предлагает план апгрейда без лишнего жаргона.

## Источник истины

Сначала найди корень **ma-hub**:
1. `~/.config/ma-hub/config` → `MA_HUB_ROOT`
2. иначе `~/ma-hub`, если есть `standards/`
3. иначе остановиться: попросить bootstrap (`git clone` публичного ma-hub + `./bootstrap/bootstrap.sh`)

Перед аудитом желательно обновить хаб до latest `main`:
`ma-hub-pull` или `$MA_HUB_ROOT/bootstrap/pull.sh`  
(если в `docs/MA-STANDARDS.md` проекта указан **Pin** — уважать pin / `MA_HUB_REF`).

Читать **целиком** (или суммировать ключевые требования):

`$MA_HUB_ROOT/standards/`

Файлы:
- `VERSION`
- `00-operating-model.md`
- `01-role-and-voice.md`
- `02-product-preferences.md`
- `03-engineering-defaults.md`
- `04-design-defaults.md`
- `05-release-and-quality.md`

## Старт

1. Убедиться, что workspace — **конкретный проект** (не home и не сам ma-hub, если цель — другой продукт). Если home — спросить путь и `move_agent_to_root` перед изменениями.
2. Прочитать текущий `docs/MA-STANDARDS.md` если есть (версия, policy track main / pin, отклонения).
3. Сравнить с `$MA_HUB_ROOT/standards/VERSION`.
4. **Найти дубли стандартов в проекте** (см. ниже) — до массовых правок UI.

## Одна визитка, без копий правил

В продукте **один** файл связи со стандартами: `docs/MA-STANDARDS.md`.  
Сами правила живут только в ma-hub (`standards/`). **Не копировать** папку `standards/` и не плодить вторые «MA rules» в проекте.

### Что искать (кандидаты на дубль)

Искать по именам и смыслу, например:
- `*STANDARD*`, `*MA-*`, `docs/*standard*`, `docs/*ma*`
- длинные копии operating / design / engineering / release правил из хаба
- Cursor rules / docs, которые повторяют процесс MA (роль, голос, релиз-чеклист хаба)

**Не трогать** чисто продуктовое: бренд, домен, доступы, ниша — если это не копия хаба. При сомнении → в отчёт как «не трогать», не удалять.

### Что делать с каждым кандидатом

Для каждого файла в отчёте предложить одно действие:

| Действие | Когда |
|----------|--------|
| **Перенести → удалить** | Файл целиком дублирует стандарты MA или устаревшую копию правил |
| **Перенести → оставить ссылку** | Файл ещё нужен (на него ссылаются) или содержит смесь: уникальное продуктовое + дубль MA |
| **Не трогать** | Чистый продукт / бренд / домен без дубля хаба |

Уникальное и нужное (отклонения, pin, локальные договорённости) → **в `docs/MA-STANDARDS.md`**.  
Дублирующий текст → убрать.  
В файле со ссылкой оставить короткий указатель: «стандарты MA — см. `docs/MA-STANDARDS.md`» (и при необходимости 1–2 строки уникального).

### Удаление — только после «ок»

**Всегда safe по умолчанию:** сначала список «удалить / ссылка / не трогать», правки файлов — только после явного согласия пользователя. Не удалять молча.  
После «ок» на план — сделай **весь согласованный пакет** правок подряд (сначала доведи, потом один итог). Не останавливайся после каждого файла с вопросом «этот файл ок?», если он уже был в утверждённом списке.

## Уточнить (≤3, с дефолтами)

1. **Глубина**  
   Рекомендую: **аудит + план + согласование, потом правки**.  
   Варианты: только отчёт / отчёт+правки safe / отчёт+правки auto.

2. **Объём**  
   Рекомендую: **продукт + дизайн + инженерия + релиз-чеклист + дедуп файлов стандартов**.  
   Варианты: только UI / только безопасность и доступ / всё.

3. **Нельзя ломать**  
   Рекомендую: **текущий прод-поток и бренд проекта сохраняем; стандарты — поверх; продуктовые правила не сносим**.  
   Попросите назвать святое.

Если ответов нет — брать дефолты и озвучить.

## Аудит (обязательный отчёт)

**Сначала** блок «Статус процесса» (`$MA_HUB_ROOT/standards/00-operating-model.md`):

| Поле | Значение |
|------|----------|
| Сейчас | нужно ваше решение / идёт процесс / готово |
| Что происходит | … |
| Ждём | ничего / ваш «ок» на план / … |
| Обычно занимает | — / пока не ответите |
| Что от вас | ничего / ответить на уточнения / «ок» на план · правки |

До «ок» на план: **нужно ваше решение**.  
После выполнения пакета: **готово** (или снова **нужно ваше решение**, если следующий пакет ждёт согласия).

Структура отчёта на русском, деловым языком:

### Вердикт
Одна фраза: насколько проект отстаёт от стандартов vX.Y.Z.

### Соответствует
Что уже хорошо (короткий список).

### Пробелы (приоритет)
- Область (дашборд / канбан / auth / дизайн / инженерия / релиз)
- Что не так по стандарту
- Риск для бизнеса
- Предлагаемое действие
- Effort: S / M / L

### Дубли стандартов в проекте
Таблица или список:
- путь файла
- что внутри (кратко)
- действие: удалить / оставить со ссылкой / не трогать
- что перенести в `docs/MA-STANDARDS.md`

### Локальные отклонения
Где проект **осознанно** другой (бренд, ниша) — не ломать, зафиксировать в `docs/MA-STANDARDS.md`.

### Git-хуки vs `/MA-deploy` (скорость релиза) — все проекты
Сверить husky / lefthook / pre-commit с `$MA_HUB_ROOT/templates/git-hooks-ma-deploy.md` (**жёсткий gate** в `/MA-deploy` Phase 0, зеркало thin CI):
- pre-commit гоняет полный `pnpm test` / vitest без `MA_ATOMIC_PACKING=1`? → **блокер** (в auto чинит `/MA-deploy`, в safe ждёт «чини»)
- Отдельный revise только ради хуков **не обязателен**, если следующий шаг — `/MA-deploy auto`
- **Не** предлагать ослабить thin CI или убрать локальный build

### План подтягивания
Пошагово, пакетами. Safe = пауза после каждого пакета. Auto = чекпоинты без лишних вопросов.  
Первый пакет желательно: **визитка + дедуп** (после «ок»), потом UI/инженерия; хуки — отдельный маленький пакет, если тяжёлые.

## Исполнение

- До «ок» на план (в safe) — **не** массово переписывать UI и **не** удалять файлы.
- При правках: следовать standards + skills (impeccable / shadcn / next / supabase / verification).
- Создать/обновить `docs/MA-STANDARDS.md` из шаблона  
  `$MA_HUB_ROOT/templates/project-ma-link.md`  
  с актуальной версией, policy (`track main` или pin), отклонениями **и** тем, что перенесли из дублей.
- После «ок» на дедуп: удалить или заменить дубли ссылкой — чтобы в проекте не копились миллионы файлов со стандартами.
- Обновить строку в `$MA_HUB_ROOT/registry/projects.md` (если доступен хаб для записи).
- Не коммитить без просьбы.
