# /MA-revise-project

Подтянуть **текущий проект** к актуальным стандартам из публичного **ma-hub**.

Вы — консультант/БА. Ассистент — senior, который аудирует и предлагает план апгрейда без лишнего жаргона.

## Источник истины

Сначала найди корень **ma-hub**:
1. `~/.config/ma-hub/config` → `MA_HUB_ROOT`
2. иначе `~/ma-hub`, если есть `standards/`
3. иначе остановиться: попросить bootstrap (`git clone` публичного ma-hub + `./bootstrap/bootstrap.sh`)

Перед аудитом желательно обновить хаб до latest `main`:
`ma-hub-pull` или `$MA_HUB_ROOT/bootstrap/pull.sh`  
(если в `docs/MA-STANDARDS.md` проекта указан **Pin** — уважать pin / `MA_HUB_REF`).

Читать **целиком** (или суммировать ключевые требования):

`$MA_HUB_ROOT/standards/`

Файлы:
- `VERSION`
- `00-operating-model.md`
- `01-role-and-voice.md`
- `02-product-preferences.md`
- `03-engineering-defaults.md`
- `04-design-defaults.md`
- `05-release-and-quality.md`

## Старт

1. Убедиться, что workspace — **конкретный проект** (не home и не сам ma-hub, если цель — другой продукт). Если home — спросить путь и `move_agent_to_root` перед изменениями.
2. Прочитать текущий `docs/MA-STANDARDS.md` если есть (версия, policy track main / pin, отклонения).
3. Сравнить с `$MA_HUB_ROOT/standards/VERSION`.

## Уточнить (≤3, с дефолтами)

1. **Глубина**  
   Рекомендую: **аудит + план + согласование, потом правки**.  
   Варианты: только отчёт / отчёт+правки safe / отчёт+правки auto.

2. **Объём**  
   Рекомендую: **продукт + дизайн + инженерия + релиз-чеклист**.  
   Варианты: только UI / только безопасность и доступ / всё.

3. **Нельзя ломать**  
   Рекомендую: **текущий прод-поток и бренд проекта сохраняем; стандарты — поверх**.  
   Попросите назвать святое.

Если ответов нет — брать дефолты и озвучить.

## Аудит (обязательный отчёт)

Структура отчёта на русском, деловым языком:

### Вердикт
Одна фраза: насколько проект отстаёт от стандартов vX.Y.Z.

### Соответствует
Что уже хорошо (короткий список).

### Пробелы (приоритет)
- Область (дашборд / канбан / auth / дизайн / инженерия / релиз)
- Что не так по стандарту
- Риск для бизнеса
- Предлагаемое действие
- Effort: S / M / L

### Локальные отклонения
Где проект **осознанно** другой (бренд, ниша) — не ломать, зафиксировать в `docs/MA-STANDARDS.md`.

### План подтягивания
Пошагово, пакетами. Safe = пауза после каждого пакета. Auto = чекпоинты без лишних вопросов.

## Исполнение

- До «ок» на план (в safe) — **не** массово переписывать UI.
- При правках: следовать standards + skills (impeccable / shadcn / next / supabase / verification).
- В конце создать/обновить `docs/MA-STANDARDS.md` из шаблона  
  `$MA_HUB_ROOT/templates/project-ma-link.md`  
  с актуальной версией, policy (`track main` или pin), отклонениями.
- Обновить строку в `$MA_HUB_ROOT/registry/projects.md` (если доступен хаб для записи).
- Не коммитить без просьбы.

## Связь с другими командами

- Релиз после подтягивания: `/MA-deploy`
- Новая фича в уже подтянутом проекте: `/MA-idea-to-plan`
