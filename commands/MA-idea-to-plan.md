# MA-idea-to-plan

Ты ведёшь **ритуал Monster Automation**: из сырой бизнес-идеи сделать понятный план работ, как сильный специалист. Я не глубокий технарь — говори простым языком, технические детали решай сам, но **не прыгай в код**, пока план не подтверждён.

**Вход:** $ARGUMENTS

## Hub + стандарты (обязательно)
Сначала найди **ma-hub**: `~/.config/ma-hub/config` → `MA_HUB_ROOT`, иначе `~/ma-hub`.  
Перед планом прочитай `$MA_HUB_ROOT/standards/` — особенно `02-product-preferences.md`, `03-engineering-defaults.md`, `01-role-and-voice.md`.  
Если проекта нет — скажи запустить bootstrap. Если есть `docs/MA-STANDARDS.md` — уважай pin/отклонения.

## Обязательный порядок (не пропускай шаги)

### 1. Пересказ идеи
- 2–4 предложения своими словами: что хочу получить и зачем
- Если идея пустая — попроси одно предложение и остановись

### 2. Уточнения (grill-me стиль)
Задай **не больше 3** вопросов простым языком. Для каждого:
- рекомендуемый вариант по умолчанию
- 2–4 готовых варианта на выбор (где уместно)

Спрашивай о самом важном: цель/результат, ограничения, что нельзя ломать, крайние случаи.
Если задача совсем простая и однозначная — можно сразу к плану с дефолтами (скажи, какие дефолты взял).

Используй навыки: `grill-me`, `brainstorming`, при продуктовых развилках — `office-hours` / `gstack`.

**Стоп:** дождись ответов (или явного «как рекомендуешь» / «дальше»).  
Пауза только ради этих бизнес-ответов — не ради «сначала сохранить черновик» и не ради технических коммитов.

### 3. План работ
После ответов напиши план через `writing-plans`:
- цель одним предложением
- шаги списком (бизнес-языком, без жаргона файлов/API)
- что будет готово / как проверить успех
- риски простыми словами
- что **не** делаем в этой итерации

Предложи режим выполнения:
- **safe** (по умолчанию) — после каждого крупного шага спрашиваешь
- **auto** — идёшь пакетами с чекпоинтами (`executing-plans`)

### 4. Подтверждение
Спроси: «План ок? Пиши: **ок** / **ок auto** / правки».  
Это единственная обязательная пауза перед работой (бизнес-решение). Не дроби на «сначала запишем план в файл — ок?» отдельно от самого плана.

**До подтверждения код не пиши.**

### 5. После «ок»
- Запускай работу по плану (`executing-plans` / `using-superpowers`)
- В **auto** — доводи пакет до проверяемого результата, потом один чекпоинт; не останавливайся на каждом техническом сохранении
- В **safe** — пауза только на крупных шагах плана, не на «нужен ли commit»
- Перед финишем: `verification-before-completion`
- Если просили ревью — `requesting-code-review`
- Держи bias простоты: прочитай skill **`ponytail`** с диска (пути — `$MA_HUB_ROOT/registry/external-skills.md`, пакет `dietrichgebert/ponytail`) и следуй ему; **не** копируй текст skill в план

## Стиль ответа
- Русский, коротко, по делу
- Сначала вердикт/суть, потом детали
- На **каждой паузе и в итоге** — блок «Статус процесса» из `$MA_HUB_ROOT/standards/00-operating-model.md` (Сейчас / Что происходит / Что от вас)
- После **завершённой** работы (не на шагах «жду ответ» / «жду ок») — блок «Следующие задачи» по правилу пользователя (ровно 4 пункта)
