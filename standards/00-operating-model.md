# 00 · Operating model (правила / навыки / команды)

**Версия стандартов:** см. `VERSION`

## Роли

- **Вы:** консультант / бизнес-аналитик. Даёте цель, приоритеты, выборы (продукт, UX, иногда технологии).
- **Ассистент:** senior-исполнитель. Закрывает пробелы в инженерии, дизайне реализации, качестве и релизе. Не ждёт, что вы напишете код.

## Три слоя Cursor

### 1. Rules — всегда
Личные правила в Cursor Settings → Rules. Сейчас у вас:
1. Уточняющие вопросы перед задачей (с дефолтами и вариантами)
2. Блок «Следующие задачи» после выполненной работы
3. Ответы на русском, думает где эффективнее

**Держим как rules** — это ваш «контракт общения» на все проекты. Не переносим в skills: навыки не обязаны срабатывать в каждом чате.

### 2. Commands `/MA-…` — по вызову
Ритуалы Monster Automation. Вы явно запускаете сценарий (идея→план, дизайн экрана, релиз, revise проекта, SEO-аудит).

**Источник правды:** репозиторий **ma-hub** → папка `commands/`.  
Локально после bootstrap/pull: `~/.cursor/commands/` — это **кэш**, не место для правок «втихую».

### 3. Skills — по задаче
Глубокие плейбуки (grill-me, writing-plans, impeccable, shadcn, vercel-react…). Команды **ссылаются** на навыки; дублировать весь текст навыка в команду не нужно.

**MA-owned skills** (оркестраторы Monster Automation, напр. `seo-geo-audit`) живут в **ma-hub** → `skills/` и ставятся скриптом `install-skills.sh`.

**Внешние skills** (Aaron, Vercel, ponytail, impeccable…) — **не** копируются в хаб. В хабе только реестр ссылок: `registry/external-skills.md` + `registry/external-skills.manifest`. Установка/обновление с upstream: `bootstrap/install-external-skills.sh` (weekly / bootstrap). При запуске `/MA-*` агент **читает** актуальный `SKILL.md` с диска (пути — в реестре).

**Cursor-native** (`review-bugbot`, `review-security`) — из `~/.cursor/skills-cursor/`, не через manifest.

### 4. Standards (эта папка в **ma-hub**) — закон продукта
Общие требования: предпочтения UI, авторизация, канбан, качество, релиз.  
Публичный источник: репозиторий **ma-hub** (локально обычно `~/ma-hub` после bootstrap).  
`/MA-revise-project` сверяет любой репозиторий с этой папкой и подтягивает пробелы.  
Политика: **всегда latest с `main`** (weekly `ma-hub-pull`); для критичных продуктов — pin tag.

## Источник правды (обязательно для ассистента)

Любое изменение команд `/MA-*`, MA-skills или стандартов:

1. Править **только** файлы в ma-hub (`commands/`, `skills/`, `standards/`, при rules — `docs/user-rules.md`)
2. Прогнать `bootstrap/install-commands.sh` и `bootstrap/install-skills.sh`
3. Проверить `bootstrap/check-local-drift.sh`
4. **Commit + push ma-hub** (или явно сказать пользователю, что без push другие проекты/ноутбуки не увидят изменения)

Подробности: `docs/hub-maintenance.md`.

## Что куда класть (шпаргалка)

| Хотите… | Куда |
|---------|------|
| Всегда так со мной разговаривать | **Rule** (+ бэкап в `docs/user-rules.md`) |
| Запустить сценарий одной кнопкой | **Command `/MA-…`** в `ma-hub/commands/` |
| MA-оркестратор (свой плейбук) | **Skill** в `ma-hub/skills/` |
| Глубокий сторонний метод (TDD, Aaron SEO, ponytail…) | **Ссылка** в `registry/external-skills.*` + install-external; не копировать тело |
| «Во всех продуктах канбан выглядит так» | **Standard** (`02-product-preferences.md` и др.) |

## Конфликты — как решаем

1. **Standard** побеждает дефолты навыков, если явно записан.  
2. **Rule** побеждает тон ответа команды, если команда молчит.  
3. В ритуале до «ок» код не пишем — даже если навык умеет кодить сразу.

## Единый статус процесса (все длинные /MA-*)

В **каждом** ответе длинной команды, где есть пауза или итог, **сначала** короткий блок (до деталей). Цель: сразу видно — ждёте вас или идёт внутренний шаг.

```markdown
### Статус процесса
| Поле | Значение |
|------|----------|
| Сейчас | идёт процесс / нужно ваше решение / готово |
| Что происходит | одна фраза простым языком |
| Ждём | ничего / CI / Vercel / ваш ответ / … |
| Обычно занимает | — / ~N мин / пока не ответите |
| Что от вас | ничего / ответить на вопросы / «ок» или правки / «чини» · «не чинить» · «отложить» |
```

**Словарь «Сейчас»:**
- **идёт процесс** — внутренний шаг или отчёт без аварии; не ждём ответа, чтобы «разрешить технику»
- **нужно ваше решение** — без ответа дальше нельзя (уточнения, направление, план, чинить ли)
- **готово** — ритуал или пакет завершён

Когда «Сейчас = идёт процесс» и есть внешнее ожидание (CI, сборка на Vercel) — поля **Ждём** и **Обычно занимает** обязательны (ориентиры: CI ~3–10 мин, Vercel production ~5–10 мин).

**Нельзя** писать «нужно ваше решение» из‑за коммитов, промежуточных сохранений, ожидания CI/Vercel или «ок по истории».  
Полнее про паузы: `05-release-and-quality.md` → «Правило пауз».
