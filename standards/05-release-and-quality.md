# 05 · Release and quality

**Версия стандартов:** см. `VERSION`

## Перед релизом (минимум)
- [ ] Сборка проходит
- [ ] Критичный пользовательский путь руками проверен
- [ ] Нет секретов в диффе
- [ ] Env на проде актуальны
- [ ] Миграции (если есть) понятны и обратимы или с планом
- [ ] Preview / staging проверен, если доступен
- [ ] Проверка простоты (`ponytail-review`) в отчёте `/MA-deploy`; при `net ≤ -80` в safe — решение до merge

## Релиз-ритуал
Использовать `/MA-deploy` — полный пайплайн (ревью, ponytail, тесты, build, merge, prod, smoke).
В Phase 1: **смотрим параллельно** (волна ревью + волна проверок), **чиним по очереди**.
**Сначала** проверки и правки по рабочему дереву; **атомарные commits** — сам агент **в конце** локального цикла (перед push), без спроса. Фиксы в середине **не** коммитить по одному.

## Правило пауз (все длинные /MA-*)

Спрашивать / останавливаться **только** когда нужно **бизнес-решение** пользователя (цель, среда, направление дизайна, «чинить или нет», удалять ли файлы).

**Не** пауза и **не** «авария»:
- промежуточная техническая упаковка (коммиты, сохранение ради процесса)
- «сначала красиво зафиксируем, потом продолжим проверки»
- вопросы про дробление коммитов / «ок по истории»

**Сначала довести** проверяемый результат → **потом** один раз упаковать/записать, если это вообще нужно для следующего шага.
В отчётах **всегда** блок «Статус процесса» из `00-operating-model.md` (сейчас / что происходит / что от вас) — отделяй «нужно ваше решение» от «идёт процесс».
Внешние skills подтягиваются из upstream по `registry/external-skills.*`, не из копий в хабе.

## После релиза
- Smoke на прод
- Краткий отчёт: что вышло, риски, что мониторить
- «Следующие задачи» — по правилу пользователя

## Версионирование стандартов
- **MAJOR** — ломающие требования (менять старые проекты обязательно)
- **MINOR** — новые предпочтения без ломки
- **PATCH** — уточнения формулировок

После bump `VERSION` — прогнать `/MA-revise-project` на приоритетных репозиториях.
